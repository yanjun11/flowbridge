# FlowBridge 技术设计文档

## 1. 引言

### 1.1 背景

#### 问题背景

当前国内企业在办公自动化方面面临以下痛点：

1. **平台孤岛**：飞书、钉钉、企微、微信等平台各自独立，数据流转需要人工复制粘贴
2. **重复劳动**：大量重复性工作（如审批后执行脚本、数据同步、通知分发）需要人工处理
3. **开发成本高**：每个自动化场景都需要写代码、测试、部署，维护成本高
4. **国际工具不适用**：Zapier 等国际工具不支持国内平台，且存在数据出境合规问题

#### 现状概述

以我们团队现有的两个项目为例：

**项目 1：changelog 自动生成服务** (`/code/pyprojects/changelog`)
- 功能：GitLab Tag Push → AI 生成 Changelog → 飞书群通知
- 问题：
  - 只支持飞书，想加钉钉/企微需要改代码
  - 触发器和动作逻辑耦合，扩展困难
  - 每个新场景都要写新代码

**项目 2：飞书审批自动补偿** (`/code/pyprojects/admin-backend`)
- 功能：飞书审批通过 → 执行补偿脚本 → 飞书通知结果
- 问题：
  - 审批表单字段变更需要改代码
  - 补偿逻辑硬编码，灵活性差
  - 无法复用到其他审批场景

#### 系统要求

- **高可用性**：工作流执行成功率 > 99%
- **低延迟**：Webhook 响应时间 < 500ms，工作流执行时间 < 30s
- **横向扩展**：支持通过增加实例数量扩展处理能力
- **避免重复造轮**：复用现有项目的飞书集成、配置管理等模块

### 1.2 目标

本设计文档旨在实现以下目标：

1. **统一工作流引擎**：提供通用的触发器-动作执行框架，支持任意平台组合
2. **插件化架构**：新增平台集成只需实现插件接口，无需修改核心代码
3. **配置驱动**：工作流通过配置定义，不需要写代码
4. **可靠执行**：支持失败重试、执行记录追踪、错误告警
5. **易于扩展**：为未来的 AI 配置、可视化编辑器预留接口

### 1.3 非目标

以下内容与本设计相关，但不在本次实现范围内：

- **AI 自然语言配置**：暂不实现"用自然语言描述工作流自动生成配置"功能
- **可视化编辑器**：暂不提供拖拽式工作流设计界面
- **多租户支持**：MVP 阶段不考虑多租户隔离
- **用户权限管理**：暂不实现细粒度的权限控制
- **分布式部署**：MVP 阶段使用单机部署，不考虑跨机房容灾
- **实时监控大盘**：暂不集成 Prometheus/Grafana 监控

### 1.4 User Story

#### 1.4.1 用例 1：Atoms 自动充值审批

**角色**：运营人员

**场景**：用户申请 Cloud & AI 额度补偿

**价值**：
- 无需手动生成报表
- 多渠道自动分发
- AI 自动生成洞察

## 2. 系统设计

### 2.1 设计介绍

FlowBridge 采用 **事件驱动 + 插件化** 的架构设计，核心思想是：

1. **解耦触发器和动作**：触发器负责接收事件，动作负责执行任务，两者通过工作流引擎连接
2. **插件化扩展**：每个平台（飞书、企微、钉钉）都是一个插件，实现统一接口
3. **配置驱动**：工作流定义存储在数据库，通过 JSON 配置描述
4. **异步执行**：使用任务队列异步执行动作，避免阻塞 Webhook 响应

**设计原则**：
- **简单优先**：MVP 阶段优先实现核心功能，避免过度设计
- **复用现有**：从 changelog 和 admin-backend 提取通用模块
- **易于测试**：每个插件独立测试，核心引擎单元测试覆盖
- **渐进增强**：为未来的 AI 配置、可视化编辑器预留扩展点

#### 2.2.2 模块说明

**API Gateway**：
- 接收外部 Webhook 请求
- 提供工作流管理 API（CRUD）
- 提供执行记录查询 API
- 验证签名、限流、日志记录

**Workflow Engine**：
- **Trigger Manager**：管理触发器注册、事件分发
- **Workflow Executor**：加载工作流定义、执行动作链、错误处理
- **Action Manager**：管理动作插件、执行动作、重试逻辑

**Trigger Plugins**：
- 实现统一的 `TriggerPlugin` 接口
- 负责解析特定平台的事件数据
- 提供 Webhook 签名验证

**Action Plugins**：
- 实现统一的 `ActionPlugin` 接口
- 负责调用特定平台的 API
- 处理错误和重试

**Database (PostgreSQL)**：
- 存储工作流定义
- 存储执行记录
- 存储插件配置

**Redis Cache**：
- 分布式锁（防止重复执行）
- 任务去重（防止 Webhook 重复推送）
- 会话缓存

### 2.4 系统模块

#### 2.4.1 Trigger Plugin 接口

```python
from abc import ABC, abstractmethod
from typing import Dict, Optional

class TriggerPlugin(ABC):
    """触发器插件基类"""

    @property
    @abstractmethod
    def trigger_type(self) -> str:
        """触发器类型标识"""
        pass

    @abstractmethod
    async def validate_config(self, config: Dict) -> bool:
        """验证配置是否合法"""
        pass

    @abstractmethod
    async def parse_event(self, raw_data: Dict) -> Optional[Dict]:
        """
        解析事件数据
        返回标准化的触发数据，如果不匹配则返回 None
        """
        pass

    @abstractmethod
    async def verify_signature(self, request_data: bytes, headers: Dict) -> bool:
        """验证 Webhook 签名"""
        pass
```

**实现示例：飞书多维表格触发器**

```python
class FeishuBitableTrigger(TriggerPlugin):
    trigger_type = "feishu_bitable"

    async def parse_event(self, raw_data: Dict) -> Optional[Dict]:
        event_type = raw_data.get("header", {}).get("event_type")
        if event_type != "bitable.record.created":
            return None

        record = raw_data.get("event", {}).get("record", {})
        return {
            "record_id": record.get("record_id"),
            "fields": record.get("fields", {}),
            "app_token": raw_data.get("event", {}).get("app_token"),
            "table_id": raw_data.get("event", {}).get("table_id")
        }
```

#### 2.4.2 Action Plugin 接口

```python
from dataclasses import dataclass

@dataclass
class ActionResult:
    success: bool
    message: str
    data: Optional[Dict] = None

class ActionPlugin(ABC):
    """动作插件基类"""

    @property
    @abstractmethod
    def action_type(self) -> str:
        """动作类型标识"""
        pass

    @abstractmethod
    async def execute(self, context: Dict, config: Dict) -> ActionResult:
        """
        执行动作
        context: 包含 trigger_data 和前序动作的结果
        config: 动作配置
        """
        pass

    @abstractmethod
    async def validate_config(self, config: Dict) -> bool:
        """验证配置是否合法"""
        pass
```

**实现示例：企微通知动作**

```python
class WecomNotifyAction(ActionPlugin):
    action_type = "wecom_notify"

    async def execute(self, context: Dict, config: Dict) -> ActionResult:
        webhook_url = config["webhook_url"]
        message = self._render_template(config["message"], context)

        async with httpx.AsyncClient() as client:
            resp = await client.post(webhook_url, json={
                "msgtype": "text",
                "text": {"content": message}
            })

        if resp.status_code == 200:
            return ActionResult(success=True, message="通知发送成功")
        else:
            return ActionResult(success=False, message=f"通知发送失败: {resp.text}")
```

#### 2.4.3 Workflow Executor（核心执行引擎）

```python
class WorkflowExecutor:
    """工作流执行引擎"""

    async def execute(self, workflow_id: str, trigger_data: Dict) -> str:
        """
        执行工作流
        返回 execution_id
        """
        # 1. 加载工作流定义
        workflow = await Workflow.get(id=workflow_id)

        # 2. 创建执行记录
        execution = await WorkflowExecution.create(
            workflow_id=workflow_id,
            trigger_data=trigger_data,
            status="running"
        )

        try:
            # 3. 构建执行上下文
            context = {"trigger": trigger_data}

            # 4. 执行动作链
            for idx, action_config in enumerate(workflow.actions):
                action_result = await self._execute_action(
                    execution.id, idx, action_config, context
                )

                # 失败则中断
                if not action_result.success:
                    await self._mark_failed(execution, action_result.message)
                    return execution.id

                # 将结果加入上下文
                context[f"action_{idx}"] = action_result.data

            # 5. 标记成功
            await self._mark_success(execution)

        except Exception as e:
            await self._mark_failed(execution, str(e))

        return execution.id
```

### 2.5 系统流程

#### 2.5.1 Webhook 触发流程

```
1. 外部系统发送 Webhook 请求
   ↓
2. API Gateway 接收请求
   ↓
3. 验证签名（调用对应 Trigger Plugin）
   ↓
4. 解析事件数据（调用 Trigger Plugin.parse_event）
   ↓
5. 查找匹配的 Workflow（根据 trigger_type 和 trigger_config）
   ↓
6. 检查去重（Redis：webhook_id）
   ↓
7. 异步执行工作流（调用 WorkflowExecutor.execute）
   ↓
8. 立即返回 202 Accepted
```

**时序图**：

```
外部系统          API Gateway       Trigger Plugin    Workflow Executor    Action Plugin
   │                  │                   │                   │                  │
   │─── Webhook ─────>│                   │                   │                  │
   │                  │                   │                   │                  │
   │                  │── verify_signature ─>│               │                  │
   │                  │<─── OK ──────────────│               │                  │
   │                  │                   │                   │                  │
   │                  │── parse_event ───────>│               │                  │
   │                  │<─── trigger_data ─────│               │                  │
   │                  │                   │                   │                  │
   │                  │── execute ───────────────────────────>│                  │
   │<─── 202 ─────────│                   │                   │                  │
   │                  │                   │                   │── execute ──────>│
   │                  │                   │                   │<─── result ──────│
   │                  │                   │                   │                  │
```

#### 2.5.2 工作流执行流程

```
1. 加载 Workflow 定义
   ↓
2. 创建 WorkflowExecution 记录（status=running）
   ↓
3. 初始化执行上下文 context = {trigger: trigger_data}
   ↓
4. 遍历 actions 列表
   ├─> 创建 ActionExecution 记录
   ├─> 调用 Action Plugin.execute(context, config)
   ├─> 如果失败：
   │   ├─> 重试（最多 3 次，指数退避）
   │   └─> 仍失败：标记 WorkflowExecution 为 failed，中断
   └─> 如果成功：
       └─> 将结果加入 context，继续下一个 action
   ↓
5. 所有 actions 执行完成
   ↓
6. 标记 WorkflowExecution 为 success
```

## 3. Milestone 分解

### Phase 1: 基础设施（Day 1）

## 4. 风险评估

### 4.1 风险与应对措施

#### 风险 1：第三方 API 调用失败

**描述**：飞书、企微等平台 API 可能因网络、限流、服务故障等原因调用失败。

**影响**：工作流执行中断，用户体验差。

**应对措施**：
- 实现指数退避重试机制（最多 3 次）
- 记录详细的失败日志
- 提供手动重试接口
- 设置超时时间（默认 30s）

#### 风险 2：Webhook 重复推送

**描述**：外部系统可能因网络问题重复推送同一个 Webhook。

**影响**：工作流重复执行，导致数据重复（如重复补偿）。

**应对措施**：
- 使用 Redis 记录 webhook_id，TTL 1 小时
- 在执行前检查是否已处理
- 返回 202 但不重复执行

#### 风险 3：长时间执行阻塞

**描述**：某些动作（如脚本执行）可能耗时很长，阻塞其他请求。

**影响**：系统响应变慢，影响其他工作流。

**应对措施**：
- Webhook 接收立即返回 202，异步执行
- 设置全局超时（30s），超时则标记失败
- 未来迁移到 Celery 实现真正的异步

#### 风险 4：配置错误导致执行失败

**描述**：用户配置的动作参数错误（如 webhook_url 无效）。

**影响**：工作流执行失败，用户不知道原因。

**应对措施**：
- 创建工作流时验证配置（调用 validate_config）
- 执行失败时记录详细错误信息
- 提供配置测试接口

#### 风险 5：数据库连接池耗尽

**描述**：高并发时数据库连接池可能耗尽。

**影响**：新请求无法获取连接，系统不可用。

**应对措施**：
- 配置合理的连接池大小（min=5, max=20）
- 使用连接池监控
- 实现请求限流

### 4.2 局限性

1. **单机部署**：MVP 阶段不支持分布式部署，单点故障风险
2. **同步执行**：使用 APScheduler，不支持跨机器任务分发
3. **无可视化编辑器**：需要通过 API 或配置文件管理工作流
4. **有限的条件逻辑**：暂不支持复杂的 if-else、循环等逻辑
5. **无监控告警**：暂不集成 Prometheus/Grafana，依赖日志排查问题

### 4.3 遗留问题与优化点

以下问题计划在后续版本中解决：

1. **分布式部署支持** (v0.2)
   - 迁移到 Celery + RabbitMQ
   - 支持多实例负载均衡
   - 实现跨机房容灾

2. **AI 自然语言配置** (v0.3)
   - 集成 Claude/GPT-4
   - 自然语言描述 → 自动生成工作流配置
   - 智能推荐动作

3. **可视化编辑器** (v0.4)
   - 拖拽式工作流设计
   - 实时预览
   - 版本管理

4. **条件分支和循环** (v0.5)
   - 支持 if-else 条件判断
   - 支持 for-each 循环
   - 支持并行执行

5. **监控告警** (v0.6)
   - 集成 Prometheus 指标
   - Grafana 监控大盘
   - 失败自动告警（邮件/飞书/企微）

6. **多租户支持** (v1.0)
   - 租户隔离
   - 权限管理
   - 配额限制
