# FlowBridge 技术预研文档

## 1. 项目概述

### 1.1 项目背景

FlowBridge 是一个面向国内市场的工作流自动化平台（iPaaS），对标 Zapier，解决以下核心痛点：

1. **国内平台支持缺失**：Zapier 不支持飞书、钉钉、企业微信等国内主流办公平台
2. **数据合规问题**：Zapier 数据存储在海外，存在数据出境合规风险
3. **中文场景理解弱**：缺少针对中文业务场景的 AI 理解能力
4. **定价不友好**：按任务量计费，对中小团队成本较高

### 1.2 核心价值主张

- **国内平台原生支持**：飞书、钉钉、企微、微信等一等公民
- **数据本地化部署**：支持私有化部署，满足合规要求
- **AI 驱动配置**：自然语言描述工作流，AI 自动生成配置
- **开源友好**：核心引擎开源，降低使用门槛

### 1.3 目标用户

1. **中小企业**：需要打通多个办公平台，自动化重复流程
2. **开发团队**：需要快速集成第三方服务，减少胶水代码
3. **运营/客服团队**：需要自动化数据流转和通知

## 2. 市场调研

### 2.1 竞品分析

| 产品 | 优势 | 劣势 | 定价 |
|------|------|------|------|
| **Zapier** | 生态成熟（8000+ 应用）<br>易用性好 | 不支持国内平台<br>数据出境<br>贵 | $19.99/月起 |
| **Make (Integromat)** | 可视化强<br>逻辑复杂度高 | 同 Zapier | €9/月起 |
| **n8n** | 开源<br>可自部署 | 国内平台支持少<br>UI 较弱 | 免费/自部署 |
| **腾讯云 iPaaS** | 国内平台支持 | 企业向<br>定价不透明 | 按量计费 |
| **阿里云 EventBridge** | 云原生<br>高可用 | 偏技术向<br>配置复杂 | 按量计费 |

**结论**：市场存在明显空缺，国内没有类似 Zapier 易用性 + 国内平台支持的产品。

### 2.2 技术趋势

1. **AI 驱动自动化**：GPT-4/Claude 等大模型可以理解自然语言意图，自动生成工作流配置
2. **低代码/无代码**：非技术人员也能配置复杂流程
3. **事件驱动架构**：Webhook + 消息队列成为主流
4. **Serverless**：降低运维成本，按需扩展

## 3. 技术选型预研

### 3.1 后端框架

#### 候选方案

| 框架 | 优势 | 劣势 | 结论 |
|------|------|------|------|
| **FastAPI** | 异步性能好<br>类型提示<br>自动文档 | 生态相对年轻 | ✅ **推荐** |
| Flask | 成熟稳定<br>生态丰富 | 同步模型<br>性能一般 | ❌ |
| Django | 全功能<br>ORM 强大 | 重量级<br>异步支持弱 | ❌ |

**选择 FastAPI 的理由**：
- 与现有项目（changelog、admin-backend）技术栈一致
- 异步支持好，适合 I/O 密集型场景（调用第三方 API）
- 自动生成 OpenAPI 文档，方便集成

### 3.2 数据库

#### 候选方案

| 数据库 | 优势 | 劣势 | 结论 |
|--------|------|------|------|
| **PostgreSQL** | 功能强大<br>JSON 支持好<br>事务可靠 | 运维成本略高 | ✅ **推荐** |
| MySQL | 成熟稳定<br>生态好 | JSON 支持弱 | ❌ |
| MongoDB | 灵活 schema<br>横向扩展好 | 事务支持弱 | ❌ |

**选择 PostgreSQL 的理由**：
- 工作流配置是 JSON 结构，PostgreSQL 的 JSONB 类型支持好
- 需要事务保证执行记录的一致性
- 团队熟悉度高

### 3.3 缓存与消息队列

#### Redis
- **用途**：分布式锁、任务去重、会话缓存
- **理由**：轻量、性能好、团队熟悉

#### 消息队列选型

| 方案 | 优势 | 劣势 | 结论 |
|------|------|------|------|
| **Celery + Redis** | 成熟<br>Python 生态好 | 配置复杂 | ✅ **推荐（生产）** |
| **APScheduler** | 轻量<br>易用 | 单机<br>不适合分布式 | ✅ **推荐（MVP）** |
| RabbitMQ | 功能强大 | 运维成本高 | ❌ |

**分阶段策略**：
- MVP：使用 APScheduler（与 changelog 项目一致）
- 生产：迁移到 Celery + Redis

### 3.4 ORM 选型

| ORM | 优势 | 劣势 | 结论 |
|-----|------|------|------|
| **Tortoise ORM** | 异步原生<br>类 Django ORM | 生态较小 | ✅ **推荐** |
| SQLAlchemy 2.0 | 成熟<br>功能强大 | 异步支持复杂 | ❌ |
| Prisma | 类型安全<br>迁移方便 | Python 支持弱 | ❌ |

**选择 Tortoise ORM 的理由**：
- admin-backend 项目已使用，团队熟悉
- 异步支持好，与 FastAPI 配合完美
- 迁移工具（Aerich）成熟

### 3.5 平台 SDK 调研

#### 飞书（Lark）
- **官方 SDK**：`lark-oapi` (Python)
- **现有经验**：admin-backend 已集成
- **支持能力**：
  - ✅ 多维表格 API
  - ✅ Webhook 事件订阅
  - ✅ 审批流程
  - ✅ 机器人消息推送

#### 企业微信
- **官方 SDK**：`wechatpy` (第三方，成熟)
- **支持能力**：
  - ✅ 群机器人 Webhook
  - ✅ 应用消息推送
  - ✅ 审批流程（需企业认证）

#### 钉钉
- **官方 SDK**：`dingtalk-sdk` (Python)
- **支持能力**：
  - ✅ 群机器人 Webhook
  - ✅ 工作通知
  - ✅ 审批流程

## 4. 架构设计预研

### 4.1 核心架构模式

采用 **事件驱动 + 插件化** 架构：

```
┌─────────────┐
│  Trigger    │ ← Webhook / 定时任务 / API 调用
│  (触发器)   │
└──────┬──────┘
       │
       ▼
┌─────────────┐
│  Workflow   │ ← 工作流引擎（核心）
│  Engine     │
└──────┬──────┘
       │
       ▼
┌─────────────┐
│  Action     │ → 飞书 / 企微 / 钉钉 / HTTP / Script
│  (动作)     │
└─────────────┘
```

### 4.2 插件化设计

**Trigger 插件接口**：
```
class TriggerPlugin(ABC):
    @abstractmethod
    async def validate_config(self, config: dict) -> bool:
        """验证配置"""

    @abstractmethod
    async def setup_webhook(self, workflow_id: str, config: dict) -> str:
        """设置 webhook，返回 webhook URL"""

    @abstractmethod
    async def parse_event(self, raw_data: dict) -> dict:
        """解析事件数据"""
```

**Action 插件接口**：
```
class ActionPlugin(ABC):
    @abstractmethod
    async def execute(self, context: dict, config: dict) -> ActionResult:
        """执行动作"""

    @abstractmethod
    async def validate_config(self, config: dict) -> bool:
        """验证配置"""
```

### 4.3 数据流设计

```
Webhook 请求
  ↓
验证签名
  ↓
解析事件 (Trigger Plugin)
  ↓
查找匹配的 Workflow
  ↓
创建 Execution 记录
  ↓
执行 Action 链 (Action Plugins)
  ↓
更新 Execution 状态
  ↓
返回结果
```

## 5. 技术风险评估

### 5.1 性能风险

**风险**：单个工作流执行时间过长，阻塞其他请求

**应对措施**：
- 使用异步任务队列（Celery）
- 设置超时机制（默认 30s）
- 长时间任务拆分为多步

### 5.2 可靠性风险

**风险**：第三方 API 调用失败导致工作流中断

**应对措施**：
- 实现重试机制（指数退避）
- 记录详细的失败日志
- 支持手动重试

### 5.3 安全风险

**风险**：Webhook 伪造、敏感信息泄露

**应对措施**：
- 所有 Webhook 必须验证签名
- 敏感配置（API Key）加密存储
- 支持 IP 白名单

### 5.4 扩展性风险

**风险**：新增平台集成需要改核心代码

**应对措施**：
- 插件化架构，新平台只需实现插件接口
- 配置驱动，不硬编码平台逻辑

## 6. MVP 范围界定

### 6.1 MVP 功能清单

**必须有（P0）**：
- ✅ 飞书多维表格 Webhook 触发
- ✅ 企微群机器人通知动作
- ✅ 工作流执行引擎
- ✅ 执行记录查询
- ✅ 基础错误处理和重试

**可以有（P1）**：
- ⭕ 飞书审批触发
- ⭕ 钉钉通知动作
- ⭕ HTTP 请求动作
- ⭕ 条件分支

**暂不做（P2）**：
- ❌ AI 自然语言配置
- ❌ 可视化编辑器
- ❌ 用户权限管理
- ❌ 多租户支持

### 6.2 技术债务清单

1. **单机部署**：MVP 阶段不考虑分布式，使用 APScheduler
2. **配置硬编码**：暂不做配置 UI，通过 API 或配置文件管理
3. **简化错误处理**：只做基础重试，不做复杂的补偿逻辑
4. **无监控告警**：暂不集成 Prometheus/Grafana
